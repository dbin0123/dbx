问题描述

1. 执行存储过程 sp_copy_trade_order_history 时，超时了， 但是存储过程还在运行中 结果页面超时30秒
```sql
CALL sp_copy_trade_order_history(1, 10000, 'tb_trade_order_history');
```

2. mysql数据库，选中要关闭的数据库，右键关闭链接，此时数据库状态显示已关闭。
再次右键 打开连接，状态显示连接成功，新建查询， 执行sql,，就会发生查询超时。
并且此时，再右键关闭连接，其状态依旧是 已连接的状态

前端的连接状态和后端的真实连接池是两套完全解耦的状态,中间没有任何同步/校验机制
分析问题并给出存在的原因

---

## 问题一分析：SQL 执行超过 30s 被自动中断（本不应中断）

### 现象
执行 `CALL sp_copy_trade_order_history(1, 10000, 'tb_trade_order_history');` 时，前端结果页面 30 秒后报超时，但数据库服务端的存储过程仍在继续执行。

### 根本原因

1. **后端的 30 秒硬超时自动中断了查询（不应中断）**
   dbx 默认查询超时为 30 秒：`pub const QUERY_TIMEOUT: Duration = Duration::from_secs(30);`（[query.rs:21](file:///d:/Developments/jetbrains/workspace/rust/dbx/crates/dbx-core/src/query.rs#L21)）。该超时通过 `tokio::time::timeout(timeout_duration, future)` 实现（[query.rs:757-769](file:///d:/Developments/jetbrains/workspace/rust/dbx/crates/dbx-core/src/query.rs#L757-L769)），超时触发后直接**丢弃（drop）正在执行的 Future**，导致：
   - 查询被强制中断，即使服务端仍在执行
   - 连接被放回池中处于脏状态
   - 被中断的连接若被复用，后续查询会阻塞在之前未完成的查询上

2. **设计问题：30s 应仅是前端超时提示，不应导致后端自动中断**
   用户期望的行为是：SQL 执行超过 30s 时，前端可以提示"查询已执行 30s，仍在运行中…"，但**不应自动中断查询**。只有用户明确点击"中断"按钮时，才应中断查询。
   
   当前后端的 `tokio::time::timeout` 机制直接 drop future，没有给用户选择的机会，属于"过度干预"。

3. **缺少用户触发的中断机制**
   虽然前端有中断按钮，但后端对应驱动的服务端取消能力缺失：
   - DuckDB 有 `interrupt_handle.interrupt()`（[query.rs:829-851](file:////d:/Developments/jetbrains/workspace/rust/dbx/crates/dbx-core/src/query.rs#L829-L851)）
   - MySQL 未实现 `KILL QUERY <connection_id>`
   - PostgreSQL 未实现 `pg_cancel_backend(pid)`
   - SQL Server 未实现 `KILL <session_id>`
   - SQLite/Rqlite/Turso/ClickHouse/ES/VectorDb/InfluxDb/Agent 同样缺失

4. **超时后连接可能被污染，导致后续查询连锁超时（全局）**
   超时后该连接上 SQL/过程仍在执行。`pool_error_action` 对不同驱动超时返回不同策略（如 MySQL 为 `Keep`、SQL Server 为 `Discard`，[query.rs:2149-2155](file:///d:/Developments/jetbrains/workspace/rust/dbx/crates/dbx-core/src/query.rs#L2149-L2155)）。对 `Keep` 的驱动，若脏连接被复用，后续查询会阻塞等待前一个过程结束，表现为"后续查询也超时"。

## 问题二分析：关闭/重开连接后状态不同步导致查询超时

### 现象
关闭数据库连接 → 状态显示已关闭 → 重新打开连接 → 状态显示连接成功 → 执行 SQL 超时 → 再次关闭连接，状态仍是"已连接"。

### 根本原因

1. **前端状态独立维护，且向后端无校验**
   `connectedIds` 是纯前端的 `ref<Set<string>>`（[connectionStore.ts:147](file:///d:/Developments/jetbrains/workspace/rust/dbx/apps/desktop/src/stores/connectionStore.ts#L147)）。`ensureConnected` 仅检查这个前端集合：`if (connectedIds.value.has(connectionId)) return;`（[connectionStore.ts:905](file:///d:/Developments/jetbrains/workspace/rust/dbx/apps/desktop/src/stores/connectionStore.ts#L905)），命中即直接返回，**不再向后端校验池是否真实存在**。

2. **"关闭数据库连接"不更新前端连接态**
   `closeDatabaseConnection()` 调用后端 `close_database_pool` 移除了对应数据库的池（[connection.rs:1155-1192](file:///d:/Developments/jetbrains/workspace/rust/dbx/crates/dbx-core/src/connection.rs#L1155-L1192)），但前端只清了树节点的 children，**没有从 `connectedIds` 中删除连接级状态**（[connectionStore.ts:881-897](file:///d:/Developments/jetbrains/workspace/rust/dbx/apps/desktop/src/stores/connectionStore.ts#L881-L897)）。所以"数据库节点显示已关闭"但"连接级状态仍为已连接"。

3. **"重新打开连接"实际是空操作**
   用户再次点击打开连接时，`ensureConnected` 看到 `connectedIds` 里仍有该 id，直接 return，**不会重建后端池**。前端显示"连接成功"是假象，后端池其实已被移除或处于失效状态。

4. **执行 SQL 超时**
   查询走到后端 `do_execute` 时，`connections.get(pool_key)` 取到的是已关闭/失效的池（或取不到），要么报 "Connection not found"，要么在失效连接上等待，最终触发各种超时。

5. **再次关闭仍显示"已连接"**
   由于 `connectedIds` 从未被清理，且关闭数据库连接的路径不触碰 `connectedIds`，所以无论关闭多少次，前端连接级状态始终显示"已连接"。只有走 `disconnect()`（连接级断开）才会执行 `connectedIds.value.delete(connectionId)`（[connectionStore.ts:865](file:///d:/Developments/jetbrains/workspace/rust/dbx/apps/desktop/src/stores/connectionStore.ts#L865)）。

---

## 两个问题的共同根因

两者都源于"客户端状态"与"服务端真实状态"的脱节：
- **问题一**：30s 后端硬超时不应存在。超时应仅是前台提示，不应自动中断查询。中断应由用户手动触发。
- **问题二**：前端状态未与后端连接池同步（缺前后端状态校验机制）。

### 修复方向建议
1. **问题一（核心）**：
   - 移除 `wait_for_query_opt` 中 30s 自动超时中断逻辑，改为仅监控/提示不中断
   - 实现按驱动的服务端取消能力——MySQL 发送 `KILL QUERY <connection_id>`、Postgres 发送 `pg_cancel_backend(pid)`、SQL Server 发送 `KILL <session_id>` 等，参照 DuckDB 路径的 `interrupt_handle.interrupt()` 模式
   - 将中断按钮与后端取消能力连接，只有用户点击时才中断
2. **问题二**：在 `ensureConnected` 中增加后端池存在性校验（如 ping 或查询池状态）；`closeDatabaseConnection` 同步清理或失效相关前端状态；建立前后端状态对账机制。
