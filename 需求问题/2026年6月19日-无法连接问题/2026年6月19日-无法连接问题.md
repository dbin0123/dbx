经过对全部数据库驱动代码的完整检查，确认无法连接这是一个**系统性问题**，影响所有数据库类型。以下是更新后的完整分析。

# 数据库连接稳定性问题分析（全驱动范围）

## 一、核心结论

**这是通病，不是 MySQL 独有**。问题根源在于连接管理框架的设计缺陷，几乎所有数据库驱动都存在以下一个或多个问题：
1. keepalive 默认关闭（全局）
2. 健康检查无超时（多数驱动）
3. 元数据操作无重连（多数驱动）
4. `refresh_connections` 覆盖不全（仅 MySQL/Postgres）
5. `close_pool_kind` 形同虚设（多数驱动）
6. 查询重连逻辑不一致（部分驱动）

### 5. `close_pool_kind` 形同虚设

[crates/dbx-core/src/connection.rs:1455-1480](file:///d:/Developments/jetbrains/workspace/rust/dbx/crates/dbx-core/src/connection.rs#L1455-L1480)：

```rust
pub async fn close_pool_kind(pool: PoolKind) {
    match pool {
        PoolKind::Mysql(p, _) => { let _ = p.disconnect().await; }  // ✅
        PoolKind::Postgres(p) => p.close(),                          // ✅
        PoolKind::Agent(client) => { /* disconnect */ }              // ✅
        PoolKind::ExternalDriver { session, .. } => { session.shutdown().await; }  // ✅
        PoolKind::Sqlite(_) => {}                                    // ❌ 无操作
        PoolKind::Rqlite(_) => {}                                    // ❌ 无操作
        PoolKind::Turso(_) => {}                                     // ❌ 无操作
        PoolKind::Redis(_) => {}                                     // ❌ 无操作
        PoolKind::MongoDb(_) => {}                                   // ❌ 无操作
        PoolKind::ClickHouse(_) => {}                                // ❌ 无操作
        PoolKind::SqlServer(_) => {}                                 // ❌ 无操作
        PoolKind::Elasticsearch(_) => {}                             // ❌ 无操作
        PoolKind::VectorDb(_) => {}                                  // ❌ 无操作
        PoolKind::InfluxDb(_) => {}                                  // ❌ 无操作
    }
}
```

**影响**：当连接池被"移除"时，底层 TCP 连接并未被主动关闭，导致：
- 半开连接残留在系统中，直到 TCP 超时（可能数小时）
- 服务端连接数泄漏
- 重建连接池时可能因服务端连接数耗尽而失败

## 四、修复建议（按优先级）

### P0 - 必须修复（解决连锁失效）

1. **为所有健康检查添加超时**
   - MySQL `get_conn_with_health_check` 的 `ping`
   - PostgreSQL `refresh_connections` 的 `get()` 和 `simple_query`
   - SQL Server `test_connection` 的 `simple_query`
   - MongoDB `test_connection` 实际使用 `_timeout` 参数

2. **启用默认 keepalive**
   - `default_keepalive_interval_secs()` 从 `0` 改为 `60`
   - 或在 `create_pool` 时强制启用

3. **`refresh_connections` 覆盖所有驱动**
   - 移除 `matches!(pool, PoolKind::Mysql(..) | PoolKind::Postgres(..))` 过滤
   - 为每个驱动实现带超时的健康检查

### P1 - 重要（解决重启才能恢复）

4. **为所有元数据操作添加重连**
   - `get_columns_core`、`list_indexes_core` 等全部用 `retry_metadata_connection` 包装
   - Redis/MongoDB ops 添加重连逻辑

5. **`close_pool_kind` 实现所有驱动**
   - SQL Server：drop client（tiberius Client drop 会关闭连接）
   - MongoDB：`client.shutdown()` 或 drop
   - ClickHouse/Elasticsearch：HTTP 客户端 drop
   - Redis：drop connection

6. **`remove_stale_connection_pool` 覆盖所有驱动**
   - 添加 PostgreSQL、ClickHouse、Elasticsearch 等的陈旧检测

7. **Windows 添加系统事件处理**
   - 监听 `RunEvent::Resumed`（系统唤醒）
   - 监听 `WindowEvent::Focused`（窗口焦点恢复）
   - 网络变化检测

### P2 - 优化

8. **统一查询重连逻辑**
   - 所有驱动的 `do_execute` 失败后统一走 `ReconnectAndRetry`
   - ClickHouse/Elasticsearch 等也添加池移除

9. **连接池配置优化**
   - 各驱动统一配置 idle_timeout < 服务端超时
   - 启用 TCP keepalive 作为补充

10. **前端自动重连**
    - `recordMetadataLoadError` 检测到连接错误时自动调用后端 `reconnect_pool`
    - 限制重试次数
